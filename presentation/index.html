The Problem

Traditional AI Approach ❌

User Request → LLM → Code → Done

✗ Context Loss — Forgets previous decisions

✗ Hallucination — Plausible but wrong code

✗ No Verification — Never checks against requirements

✗ Inconsistency — Different results each run

✗ Data Risk — Uncontrolled where your code goes

Why It Fails

  • Single model for all tasks (coding ≠ reviewing)
  • No reasoning phase before implementation
  • Output never verified against spec
  • No memory between runs
  • Hard-coded to one vendor — no compliance control

Auto-Discovery Priority

# Resolution order (automatic — first available wins):
anthropic → bedrock → vertex → openai → local → github_models

# Force a specific provider for ALL skills:
LLM_PROVIDER=bedrock   # in .env

Set one or many providers. The registry discovers what's available and routes each skill to the best backend automatically.

Smart Tier-Based Fallback

When a model isn't available, the registry matches by tier, not blindly downgrades:

# Only GITHUB_TOKEN set (no Claude key):
gather_requirements (claude/balanced) → github_models/gpt-4o
analyze_repo        (claude/fast)     → github_models/gpt-4o-mini
review_code         (claude/powerful) → github_models/Llama-3.1-405B  ← tier-matched!

# With LOCAL_LLM_URL=http://localhost:11434/v1:
review_code         (claude/powerful) → local/deepseek-v2.5:latest     ← local first!

Data Privacy Decision Tree

🏢 Company data / NDA code?

→ Use bedrock or vertex (stays in your cloud)
→ Or local (stays on your machine)

🔓 Open source / personal projects?

→ github_models is free and works immediately

🧠 Need real Claude for best reasoning?

→ anthropic (direct) or bedrock/vertex (private)

Local LLMs — 100% Offline

Works With Any OpenAI-Compatible Server

# Ollama (most popular)
LOCAL_LLM_URL=http://localhost:11434/v1

# LM Studio
LOCAL_LLM_URL=http://localhost:1234/v1

# vLLM / llama.cpp / LocalAI / text-gen-webui
LOCAL_LLM_URL=http://localhost:8000/v1

# Explicitly list models (or let the provider auto-discover):
LOCAL_LLM_MODELS=deepseek-coder-v2:latest,llama3.1:70b,codestral:latest

Example: DeepSeek on Ollama

# 1. Pull and run DeepSeek locally
ollama pull deepseek-coder-v2:latest
ollama serve

# 2. Add to .env — that's it
LOCAL_LLM_URL=http://localhost:11434/v1
LLM_PROVIDER=local        # optional: force all skills to use local

# 3. Agent auto-discovers models
agent status
#  ✅ local   http://localhost:11434/v1 (3 models available)
#  ── github_models  GITHUB_TOKEN (fallback)

agent review < myfile.py   # runs through deepseek locally

Zero external network calls. Data never leaves your machine.

Supported Local Models

Code-focused: DeepSeek Coder V2, Codestral, StarCoder2, Code Llama

General reasoning: Llama 3.1 (8B / 70B / 405B), Mixtral 8×22B

The Critic Loop

Self-Correction in Action

Generate Result
    ↓
Run Critic (best available model)
    ↓
Score < 90? → Improve Result → Loop (max 5)
    ↓
Score >= 90? → Accept and Move Forward
    ↓
Verify Against Spec

Real Example: Architecture Design

Iteration 1: Score 62 ❌ Missing validation layer

Iteration 2: Score 85 ❌ Inconsistent naming

Iteration 3: Score 94 ✓ ACCEPTED

Different reasoning pass prevents reinforcing mistakes

Best Practices

✅ DO

  • Separate concerns — reasoning, coding, reviewing are different tasks
  • Use critic loops — never ship without 90/100 quality score
  • Verify specs — always validate against requirements
  • Persist memory — store architectural decisions
  • Choose the right provider for your data sensitivity
  • Use LLM_PROVIDER=local for confidential codebases

❌ DON'T

  • Use the same model for all tasks
  • Trust outputs without verification
  • Hard-code a single LLM vendor
  • Ship first output without critic loop
  • Send company code to external APIs without checking your data policy
